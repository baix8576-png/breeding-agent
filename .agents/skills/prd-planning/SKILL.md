name: prd-planning
description: 面向 orchestrator 的总控规划技能，用于将模糊需求拆解为可路由、可验收、可发布的跨角色任务体系。

## 触发条件
- 用户提出新功能方向、版本目标、跨角色协作或阶段里程碑需求。
- 需求不完整或存在多种实现路径，需要先做范围收敛。
- 需要明确阶段 Gate（设计通过、实现开始、测试开始、可发布）。

## 不应触发的场景
- 任务已是单角色、低风险、边界清晰的局部改动。
- 仅执行固定脚本或纯格式修改，不涉及规划与路由。

## 输入
- 用户需求原文与业务目标。
- 项目约束（时间、资源、环境、安全、合规）。
- 现有文档与上下文（README、AGENTS、架构说明、已有决策）。

## 输出
- 需求结构化拆解：`目标 / 约束 / 风险 / 交付物 / 里程碑`。
- 任务分解表（含任务归属角色、依赖顺序、Gate 目标）。
- 阶段决策清单（当前阶段是否可进入下一 Gate）。
- 在角色专属输出之外，补充统一公共回传头：`role / task_id / scope_in / scope_out / risks / next_actions / ready_for_gate`

## 执行步骤
1. 抽取需求五要素  
   将模糊需求拆成以下结构：
   - 目标：业务问题、成功标准、范围边界。
   - 约束：时间、算力、人员、平台、数据边界、合规要求。
   - 风险：技术不确定性、依赖风险、安全与质量风险。
   - 交付物：文档、配置、脚手架、评估报告、发布结论。
   - 里程碑：阶段目标与时间顺序（按 Gate 驱动）。
2. 输出任务分解表  
   使用统一字段输出：
   - `task_id`
   - `task_name`
   - `owner_role`
   - `objective`
   - `input`
   - `output`
   - `dependency`
   - `gate_target`
   - `acceptance`
3. 指定子任务归属角色  
   归属规则：
   - 架构边界/接口契约 -> `architect`
   - 编排与工具路由 -> `llm_orchestrator`
   - 群体遗传/数量遗传流程 -> `popgen_quantgen`
   - 集群调度/资源策略 -> `hpc_scheduler`
   - 安全门禁/熔断 -> `safety_fuse`
   - 测试基准/发布门槛 -> `test_eval`
4. 识别依赖顺序  
   依赖判定规则：
   - 设计先于实现，接口先于编码，安全基线先于外部调用，测试准入先于发布。
   - 若 A 的输出被 B 消费，则 A -> B。
   - 若任务可并行，标记为 `parallel`，并定义汇合点。
   - 识别关键路径并标记阻塞任务。
5. 输出阶段 Gate 结论  
   每阶段必须给出通过/不通过与缺口：
   - `Gate 1 设计通过`：边界清晰、接口可用、风险可控。
   - `Gate 2 实现开始`：设计冻结、输入输出明确、负责人到位。
   - `Gate 3 测试开始`：功能闭环可运行、日志可追溯、测试数据可准备。
   - `Gate 4 可发布`：测试通过、安全门禁通过、文档与审计记录完整。

## 失败处理
- 若目标冲突：输出冲突矩阵（冲突点、影响、方案A/B、推荐方案、弃选理由）。
- 若信息缺失：输出最小补充项与默认假设，先推进低风险部分。
- 若依赖不明：建立临时占位接口并标注“阻塞解除条件”。
- 若角色越权：回退到 orchestrator 重新路由，不允许直接跨界实现。

## 与其他角色的协作方式
- 与 `architect`：先确认系统边界和接口契约，作为 Gate 1 核心输入。
- 与 `llm_orchestrator`：确认自然语言到工具链路的编排闭环。
- 与 `popgen_quantgen`：确认分析流程规范、输入输出与解释模板。
- 与 `hpc_scheduler`：确认作业提交、资源估算、轮询与故障恢复策略。
- 与 `safety_fuse`：确认上云字段、脱敏、熔断与人工审批门禁。
- 与 `test_eval`：确认测试矩阵、回归策略、发布阻断条件。

## 示例：orchestrator 典型拆解输出格式
需求示例：开发一个面向羊群基因组选择分析的多角色智能系统。

### 1) 需求五要素拆解
- 目标：支持羊群基因组选择分析的端到端智能协作，覆盖需求理解、流程编排、作业执行、安全门禁与测试评估。
- 约束：本地数据不出域；首版仅 SLURM；先交付骨架不实现算法；需可扩展到更多流程。
- 风险：角色边界重叠、编排链路不稳定、资源估算偏差、安全策略遗漏、测试覆盖不足。
- 交付物：agent/skill 配置骨架、路由规则、阶段 Gate 清单、测试与发布准入标准。
- 里程碑：Gate 1 设计通过 -> Gate 2 实现开始 -> Gate 3 测试开始 -> Gate 4 可发布。

### 2) 任务分解表（示例）
| task_id | task_name | owner_role | objective | input | output | dependency | gate_target | acceptance |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| T1 | 架构边界与接口草案 | architect | 明确模块边界与契约 | 目标/约束 | 边界文档与接口清单 | none | Gate 1 | 边界无冲突、接口可交接 |
| T2 | LLM 编排流程草案 | llm_orchestrator | 明确意图路由与工具调用 | T1 输出 | 编排流程与异常回退 | T1 | Gate 1 | 关键路径闭环成立 |
| T3 | 群体/数量遗传流程规范 | popgen_quantgen | 定义分析流程输入输出 | T1 输出 | 流程模板与产物规范 | T1 | Gate 1 | 流程可编排、口径一致 |
| T4 | SLURM 调度策略 | hpc_scheduler | 定义资源估算与状态管理 | T2/T3 输出 | 调度接口与状态机 | T2,T3 | Gate 2 | 作业链路可执行 |
| T5 | 安全与熔断门禁 | safety_fuse | 定义脱敏与熔断策略 | T2/T4 输出 | 安全规则与审批点 | T2,T4 | Gate 2 | 高风险路径可阻断 |
| T6 | 测试与发布门槛 | test_eval | 定义验证矩阵与发布准入 | T1-T5 输出 | 测试计划与 Gate 判据 | T1,T2,T3,T4,T5 | Gate 3/4 | 具备发布判定能力 |

### 3) Gate 结论模板（示例）
- Gate 1 设计通过：`通过/不通过`；缺口：`...`；责任角色：`...`。
- Gate 2 实现开始：`通过/不通过`；缺口：`...`；阻塞项：`...`。
- Gate 3 测试开始：`通过/不通过`；缺口：`...`；测试准备状态：`...`。
- Gate 4 可发布：`通过/不通过`；缺口：`...`；发布建议：`...`。
