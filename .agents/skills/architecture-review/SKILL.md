name: architecture-review
description: 面向 architect 的架构评审技能，用于在实现前或重大改动前检查模块边界、接口契约、依赖方向、扩展点和设计 Gate 就绪度。

## 触发条件
- 新增模块、跨模块改动、核心接口变更。
- 出现角色边界混乱、依赖循环或职责重叠风险。
- 需要在“设计通过 / 实现开始”之间给出架构 Gate 结论。
- 需要对长期演进成本、配置复杂度或扩展难度进行提前评估。

## 不应触发的场景
- 纯文案修改或无架构影响的小改动。
- 只涉及单文件注释与格式调整，不影响接口语义时。
- 已有清晰架构结论，当前只是按既定设计补齐局部实现时。

## 输入
- 当前架构描述、目录结构、接口草案。
- 需求范围与非目标约束。
- 风险记录与历史设计决策（如有）。
- 涉及的模块清单、状态流或数据流说明（如有）。

## 输出
- 架构评审结论（通过/有条件通过/驳回）。
- 风险列表与整改建议。
- 最小化改动路径与后续演进建议。
- 在角色专属输出之外，补充统一公共回传头：`role / task_id / scope_in / scope_out / risks / next_actions / ready_for_gate`
- 结构化评审结果，至少包含：
- `review_scope`
- `in_scope`
- `out_of_scope`
- `current_structure`
- `proposed_structure`
- `key_interfaces`
- `design_constraints`
- `risks`
- `open_questions`
- `decision`
- `ready_for_gate`

## 执行步骤
1. 界定评审范围  
   明确本次变更影响的模块、目录、接口、状态流和协作角色。
2. 检查边界与职责  
   判断模块职责是否单一，是否出现重复能力、角色越权或边界模糊。
3. 检查接口契约  
   检查输入、输出、错误语义、状态语义、配置项和扩展方式是否清晰。
4. 检查依赖方向  
   检查是否存在双向依赖、隐式耦合、共享状态蔓延、跨层直连或后续难以替换的点。
5. 检查扩展与演进能力  
   判断未来新增分析流程、新增调度器、新增知识源时是否需要大规模改动。
6. 检查安全与可测性前置条件  
   确认高风险路径是否可被 `safety_fuse` 拦截，关键接口是否可被 `test_eval` 验证。
7. 给出 Gate 结论  
   输出是否允许进入实现阶段，以及必须先补齐的缺口。

## 失败处理
- 若文档不足以评审：先输出缺口模板并要求补齐关键字段。
- 若改动范围过大：拆分为可审查批次，避免一次性风险过高。
- 若角色越权：标注越权点并退回给 orchestrator 重新分派。
- 若接口未冻结：结论应为“不通过 Gate”，要求先补契约草案。
- 若关键依赖不明：先建立占位接口和责任归属，再继续评审。

## 与其他角色的协作方式
- 与 `orchestrator` 同步评审结论和里程碑影响。
- 与 `llm_orchestrator` 对齐编排链路的接口边界和状态流。
- 与 `popgen_quantgen` 对齐领域流程模块的输入输出与责任边界。
- 与 `hpc_scheduler` 对齐作业接口、资源语义和日志归档接口。
- 与 `safety_fuse` 联合审查高风险架构路径。
- 与 `test_eval` 对齐可测性和验收挂钩点。

## 评审重点清单
- 模块是否按职责单一原则拆分。
- 目录结构是否能反映职责边界。
- 接口是否稳定且具备错误语义。
- 是否优先采用 manifest / 配置驱动，而不是把扩展点写死。
- 是否为新增分析流程和新增调度器预留标准入口。
- 是否能支持审计、回滚、熔断和测试挂钩。

## Gate 判断规则
- `设计通过`：
  - 边界清晰，接口可交接，关键风险已知且可控。
- `有条件通过`：
  - 主结构合理，但存在少量待补字段或文档缺口，不阻断总体方向。
- `不通过`：
  - 边界模糊、依赖方向错误、接口不清或存在明显越权与高风险路径。
