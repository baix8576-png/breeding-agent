name: llm-workflow-design
description: 面向 llm_orchestrator 的 workflow 设计技能，用于定义 prompt 分层、上下文管理、工具调用、RAG、结构化输出、错误恢复与成本控制。

## 触发条件
- 需要设计或重构自然语言到执行计划的链路。
- 需要明确本地知识优先与外部检索兜底策略。
- 需要定义失败回退、重试和解释输出机制。
- 需要将结果标准化为可测试、可审计、可回归的结构化输出。
- 需要为长上下文任务建立摘要、分块和缓存方案。

## 不应触发的场景
- 仅做集群参数调优，不涉及编排策略时。
- 仅做领域流程细节调整，不涉及路由与调用策略时。
- 仅做高风险工具调用审批或熔断裁决时。
- 已有稳定 workflow，只是在既定流程中补写局部业务实现时。

## 输入
- 用户请求样例与业务约束。
- 可用工具列表与工具接口定义。
- 知识库结构与安全边界策略。
- 模块边界、结构化输出需求和评估需求（如有）。

## 输出
- `workflow_steps`：编排流程规范（步骤、条件、分支、回退）。
- `prompt_layers`：system / role / task / data context 分层设计。
- `tool_calling_rules`：工具调用前置校验、调用条件、失败降级。
- `rag_strategy`：知识检索、证据组织、引用与缓存策略。
- `output_schema`：结构化输出建议。
- `recovery_policy`：fallback / retry / timeout 方案。
- `cost_guardrails`：模型选择、token 预算、上下文压缩建议。
- `integration_notes`：与 architect / safety_fuse / test_eval 的接口说明。
- 在角色专属输出之外，补充统一公共回传头：`role / task_id / scope_in / scope_out / risks / next_actions / ready_for_gate`

## 执行步骤
1. 定义 workflow 范围
   明确任务是单轮分析、工具调用型还是检索增强型，并标注 In/Out of Scope。
2. 设计 prompt 分层
   将以下内容分离：
   - `system prompt`：全局行为边界、真实性约束、输出纪律。
   - `role prompt`：当前角色职责和越权禁止。
   - `task prompt`：本次任务目标、输入、输出和约束。
   - `data context`：检索结果、工具返回、日志摘要、配置元数据。
3. 设计上下文管理
   对长上下文任务提出：
   - 摘要压缩策略
   - 分块策略
   - 缓存策略
   - 引用回填策略
   - 无关上下文剔除策略
4. 设计工具调用策略
   在调用前必须检查：
   - 输入是否完整
   - 参数是否满足最小必填项
   - 是否需要人工确认
   - 是否涉及高风险动作
5. 设计 RAG 流程
   采用“本地知识优先、外部检索兜底、证据合并后再生成”的原则，并保留证据来源字段。
6. 设计结构化输出
   输出必须尽量结构化，优先定义字段名、状态字段、错误字段、来源字段和下一步动作字段。
7. 设计错误恢复
   为每类工具定义失败后的降级路径、重试上限、超时处理和人工接管条件。
8. 设计成本控制
   定义模型分层、token 预算、摘要阈值、缓存复用和多轮对话截断策略。
9. 输出 workflow 方案
   交付可供 architect、safety_fuse、test_eval 复核的结构化设计结果。

## 失败处理
- 若工具接口不完整：回传缺失字段给 architect /对应工具负责人。
- 若存在安全冲突：立即移交 safety_fuse 审查后再推进。
- 若流程过长不可维护：拆分成主流程与子流程模板。
- 若输出不可测试：先补结构化 schema 与状态字段，再继续设计。
- 若检索证据不足：禁止模型直接补造事实，应回退为“证据不足”状态。

## 与其他角色的协作方式
- 与 `architect`：
- 对齐模块边界、工具接口、状态流和上下文存取边界。
- 若 workflow 侵入实现层，退回由 architect 重划边界。
- 与 `safety_fuse`：
- 对齐高风险工具调用、脱敏规则、人工确认门禁和熔断前置信号。
- 将所有风险操作的自动调用路径交由 safety_fuse 复核。
- 与 `test_eval`：
- 对齐 output schema、日志字段、评测集输入格式、回归断言和失败样例。
- 确保 workflow 可被稳定评测，而不是依赖自由文本人工判读。
- 与 `popgen_quantgen` 对齐领域参数语义。
- 与 `hpc_scheduler` 对齐执行参数和状态回传格式。

## 专门规则
- 将系统提示、角色提示、任务提示与数据上下文严格分离，不允许混成单段提示。
- 工具调用前必须检查输入完整性，缺失字段时应先请求补全或回退，而不是猜测。
- 对每类工具调用规定降级策略：
- 纯只读查询工具失败：回退为“无法获取最新信息”，可尝试一次替代来源。
- 本地分析工具失败：返回错误摘要与建议，不得伪造分析完成。
- HPC 调度工具失败：返回提交失败或状态未知，不得伪造 job id、队列状态或结果文件。
- 检索工具失败：回退到本地知识或返回“证据不足”，不得补造外部资料。
- 输出必须尽量结构化，至少包含状态、证据、下一步建议和未决问题。
- 长上下文任务必须提出摘要、分块和缓存建议，优先压缩历史轮次与冗长日志。
- 在“遗传分析 + HPC 调度”场景中，模型不得直接伪造分析结果、统计值、作业状态、运行日志或产物路径。

## 模板 1：单轮分析型 workflow
- 适用：无需工具调用，仅对需求进行解释、规划或格式检查。
- 流程：
- 解析任务 -> 识别约束 -> 生成结构化回答 -> 标注不确定性。
- 输出重点：
- `summary`
- `assumptions`
- `constraints`
- `next_actions`

## 模板 2：带工具调用的 workflow
- 适用：需要先规划，再调用本地工具或调度工具。
- 流程：
- 解析任务 -> 校验输入完整性 -> 选择工具 -> 调用工具 -> 校验工具返回 -> 生成结构化结论。
- 输出重点：
- `selected_tools`
- `tool_inputs`
- `tool_results`
- `status`
- `recovery_actions`

## 模板 3：带检索与结构化输出的 workflow
- 适用：需要本地知识优先、外部检索兜底，并输出统一 schema。
- 流程：
- 解析任务 -> 检索本地知识 -> 证据不足时外部检索 -> 合并证据 -> 生成结构化结果 -> 输出引用与未决问题。
- 输出重点：
- `evidence`
- `sources`
- `decision`
- `confidence`
- `open_questions`

## 接口说明
- 与 `architect`：
- 输入：模块边界、接口契约、状态流限制。
- 输出：workflow 对接口的依赖清单、上下文读写需求、状态回传需求。
- 与 `safety_fuse`：
- 输入：高风险动作清单、allow/deny 字段、熔断前置条件。
- 输出：风险工具调用路径、人工确认点、自动降级与停止条件。
- 与 `test_eval`：
- 输入：评测集格式、回归断言、质量门槛。
- 输出：结构化 output schema、可观测日志字段、失败样例和判定标准。
