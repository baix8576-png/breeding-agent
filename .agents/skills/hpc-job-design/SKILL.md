name: hpc-job-design
description: 面向 hpc_scheduler 的作业设计技能，用于定义 SLURM/PBS 任务封装、资源估算、状态管理、日志契约和调度链路回传格式。

## 触发条件
- 需要定义任务提交、轮询、重试、超时和失败恢复策略。
- 需要规范作业脚本模板结构、dry-run 行为与调度适配边界。
- 需要建立资源申请、状态回传和日志追踪规则。

## 不应触发的场景
- 仅做领域分析逻辑调整，不涉及调度系统时。
- 仅做提示词编排优化，不涉及执行链路时。
- 仅做安全审批裁决，不涉及调度设计时。

## 输入
- 作业类型和预估资源需求。
- 调度系统约束（SLURM/PBS）。
- 安全门禁规则和人工确认策略。
- 流程阶段信息、输入数据规模、输出目录约定和日志需求。

## 输出
- `job_design`：作业设计规范（提交、查询、取消、重投、dry-run）。
- `resource_plan`：资源估算规则与默认参数占位。
- `state_model`：运行状态机与阶段转换定义。
- `log_contract`：日志追踪字段、作业索引和产物路径约定。
- `integration_notes`：与 orchestrator / safety_fuse / test_eval 的接口说明。

## 执行步骤
1. 定义调度适配层接口与状态语义。
2. 定义资源估算与提交前校验逻辑。
3. 定义轮询策略、超时策略和失败重试策略。
4. 定义产物路径、日志归档规范和 dry-run 输出格式。
5. 定义统一回传字段，至少覆盖 `role / task_id / scope_in / scope_out / risks / next_actions / ready_for_gate`。
6. 输出交付文档并请求 safety/test 复核。

## 失败处理
- 若调度约束不明确：回收配置缺口并使用保守默认占位。
- 若资源波动大：定义分级策略和人工确认阈值。
- 若失败模式不完整：先覆盖高频失败并建立扩展挂钩点。
- 若日志字段不足：先补齐作业 ID、状态、时间戳、退出码和产物路径索引，再允许进入测试。
- 若 dry-run 不可用：默认不放行高风险批量任务。

## 与其他角色的协作方式
- 与 `orchestrator` 对齐统一回传头和阶段 Gate 语义。
- 与 `popgen_quantgen` 对齐作业类型和资源画像。
- 与 `safety_fuse` 对齐危险命令拦截与熔断联动。
- 与 `test_eval` 对齐调度链路回归测试场景。
- 与 `architect` 对齐调度适配层边界和日志接口。
